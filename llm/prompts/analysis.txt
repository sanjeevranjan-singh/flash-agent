---------------------------------------------------------------------------------------
YOUR ANALYSIS TASK
---------------------------------------------------------------------------------------

Examine the system state. You must produce:

1. ENVIRONMENT STATE
   - Identify namespace / cluster from the data.
   - Determine health_status: Healthy | Degraded | Critical based strictly on
     what the data shows (restarts, error logs, warning events, non-Running pods).

2. IDENTIFIED ISSUES
   For each distinct symptom you can confirm from the data:

   - issue_name: concise identifier derived from evidence
     e.g. "oom-kill", "crash-loop-backoff", "readiness-probe-failure", "pod-eviction"
     NOT generic names like "pod-restart-loop" unless you have no other evidence.
   - severity: critical (service down / data loss risk) | warning (degraded) | info
   - affected_component: exact pod or deployment name from the data
   - status: active | resolved | unknown
   - first_seen: use the Timestamp field from the data header as a reference; "unknown"
     if no timestamp in the evidence for this specific issue
   - detected_at_sec:
       Compute as: seconds between the data Timestamp and the earliest log line or
       event that confirms this issue. Use 0 if the issue is already present at the
       data Timestamp. Use null ONLY if there is truly no timestamp reference at all.
   - mitigated_at_sec:
       Compute from log/event timestamps if a recovery is visible. null if still active.

   - root_cause:
       Use ONLY explicit evidence:
         * exact log lines showing the failure reason (OOMKilled, signal 9, panic, etc.)
         * Kubernetes warning event reasons (BackOff, OOMKilling, FailedScheduling)
         * restart counts combined with log crash context
       Write the actual evidence, e.g.:
         "OOMKilled: container exceeded memory limit (log: 'Killed process 1 ...')"
       NOT a generic statement.
       Only use "insufficient data to determine root cause from provided logs" if
       there are truly zero log lines and zero warning events for this pod.

   - impact_summary: service availability or performance impact inferred from data.
   - remediation_steps: specific actionable steps referencing the actual pod/component.

3. SYSTEM ERRORS
   For each pod with an actual error log excerpt visible in the data:
   - component_name, error_pod, error_message (exact 1-2 line log excerpt)
   - root_cause (from log evidence), remediation

4. OVERALL HEALTH SUMMARY
   - Instance counts, error/warning counts, resource metrics if present
   - overall_health_score 0-100:
       100 = fully healthy, no issues
       80-99 = minor issues (info severity, transient)
       60-79 = warning-level issues present
       40-59 = multiple warnings or one degraded component
       0-39 = critical failures or service unavailability

CRITICAL RULES
---------------------------------------------------------------------------------------

2. Derive all findings from logs, events, restart counts, and resource states only.
3. "insufficient data" is reserved for the case where the entire data payload
   is empty (no events, no pods, no metrics). If you can see ANY pod row,

EVIDENCE CHANNELS (universal — work for any application, any fault, any cluster)
---------------------------------------------------------------------------------------
You are a continuous-monitoring agent. You do NOT know in advance what the
application is, what limits the operator configured, or what fault (if any)
will occur. You decide whether the system is healthy by looking at what is
observably anomalous in the data right now. Treat the following as your
primary evidence channels — each one is independent, and ANY ONE of them
firing is sufficient to raise an issue:

A. POD STATE
   - Any pod not in phase Running with all containers Ready is anomalous.
   - Phases CrashLoopBackOff, Error, ImagePullBackOff, ErrImagePull, Pending
     (for more than a brief startup window), Terminating (stuck), Unknown
     all warrant an issue entry.

B. POD RESTART COUNTS
   - restart_count > 0 on any pod is direct evidence that the container
     previously failed. Even if the pod is now Running, the restart is a
     past incident and should be reported (status="resolved" if no recent
     warning events, "active" if BackOff/Killing events are still occurring).

C. POD AGE SKEW
   - If a pod's age is seconds-to-minutes while sibling pods in the same
     deployment / namespace are hours-to-days old, that pod was just
     (re)created. This is the primary signal for pod-deletion or eviction
     events where the restart_count of the new pod resets to 0. Compare
     ages within the data and flag any outlier.

D. KUBERNETES WARNING EVENTS
   - Any event with type=Warning is anomalous. Reasons that always indicate
     a real incident: Killing, OOMKilled, BackOff, CrashLoopBackOff,
     FailedScheduling, FailedMount, FailedAttachVolume, Evicted, NodeNotReady,
     Unhealthy (probe failure), FailedCreatePodSandBox, NetworkNotReady,
     NodeHasDiskPressure, NodeHasMemoryPressure.
   - Each distinct (pod, reason) pair from the EVENTS section should produce
     an identified_issues entry, using the event reason as the issue_name
     evidence and the event message as the root_cause evidence.

E. APPLICATION LOG ERROR PATTERNS
   - panic, fatal, OOMKilled, "signal 9", "signal: killed", segmentation
     fault, "connection refused", "connection reset", "no route to host",
     "i/o timeout", "deadline exceeded", "context canceled", "dial tcp ...
     refused", 5xx response codes, "database is down", "circuit breaker
     open", uncaught exception traces. Quote the exact log line as
     root_cause.

F. PROMETHEUS METRIC ANOMALIES (when METRICS section is present)
   - prometheus_targets_up: any value < total expected targets means a
     scrape target is down (the application or its sidecar is unreachable).
   - restarts_per_pod: any pod with a non-zero value is direct evidence of
     past failure (cross-check with pod restart count). When the rendered
     line says "no recent crash-loop / OOMKill", treat that as confirmed
     negative — saturation without restarts is a resource-stress fault,
     not a crash.
   - cpu_per_pod / memory_per_pod: these are RAW USAGE values in cores /
     bytes, rendered with `value / limit (NN% of limit)` when the pod
     declares a limit. Treat as anomalous when:
       * saturation reaches >=80% of the declared limit (resource ceiling),
       * a single pod's usage is dramatically higher than its peers in
         the same namespace (order-of-magnitude outlier),
       * usage trends sharply against the historical baseline of the same
         pod (when prior values are present in the data),
       * memory grows monotonically (memory leak signature).
     If neither outlier, ceiling, nor trend is present, do not invent a
     saturation issue.
   - cpu_throttle_per_pod: a non-zero value is the kernel actively capping
     a container at its CPU limit — direct evidence of CPU stress that is
     visible even when the rate(usage) average smooths the spike out. If
     a pod is throttled, flag CPU-stress with the throttled pod as
     affected_component regardless of where it ranks in cpu_per_pod_top.
   - network_rx_per_pod / network_tx_per_pod: a pod whose RX or TX rate
     drops to zero while peers continue to communicate is observably
     partitioned (network fault). RX-only or TX-only zero is asymmetric
     loss; both zero is full isolation.
   - fs_usage_per_pod_top_mb: a pod whose filesystem usage is an
     order-of-magnitude peer outlier, or that grows monotonically across
     scans, indicates a disk-fill / log-flood / tmpfs-exhaustion fault.
   - "Top PVC / volume usage" rows tagged [NEAR-FULL] (>=80%) are direct
     evidence of disk-fill on PVC-backed volumes, independent of cAdvisor
     fs labels. Treat as a high-severity disk fault.
   - "Pods stuck in waiting state" rows are kubelet-emitted status, fire
     regardless of resource limits. Reasons map to faults:
       * CrashLoopBackOff → flag crash-loop-backoff (Critical)
       * ImagePullBackOff / ErrImagePull → flag image-pull-failure
       * ContainerCreating beyond ~30s → flag stuck-creating
       * ContainerStatusUnknown → flag node/runtime issue
   - "Pods with non-Completed last-terminated reason" rows:
       * OOMKilled → flag oom-kill / memory-saturation (Critical) — this
         signal is present even when no memory limit is declared, because
         the kernel records the kill regardless.
       * Error → flag container-error-exit
   - Lines tagged [PEER-OUTLIER N.Nσ] mark pods whose CPU or memory is
     N standard deviations above their namespace peers. This is direct
     evidence of resource-stress that DOES NOT require declared limits or
     visible kernel throttling. Flag the outlier pod with cpu-saturation
     or memory-saturation. Sigma >= 3σ should be Critical, 2-3σ Warning.

G. POD COUNTS
   - desired vs ready replica mismatch in any deployment is anomalous.
   - any non-zero count of pods in phases Failed / Unknown / Pending
     (beyond brief startup) is anomalous.

EVIDENCE-AGGREGATION RULES
   - Each evidence channel is independent. ANY single channel firing is
     enough to set health_status = Degraded and add an identified_issues
     entry. Multiple channels firing on the same pod = Critical.
   - Do NOT require corroboration across channels. A Killing event alone
     is enough. A non-zero restart_count alone is enough. An order-of-
     magnitude CPU outlier alone is enough.
   - If at least one identified_issues entry exists, overall_health_score
     MUST be < 100. The mapping defined above (60-79 warning, 40-59 multiple
     warnings, 0-39 critical) is the ONLY scoring scheme — do not return
     75 with zero issues; that combination is contradictory.
   - "insufficient data" is reserved for the case where the entire data
     payload is empty (no events, no pods, no metrics). If you can see
     pods at all, you can reason about them.

You are not asked to identify the fault that was injected (there may not
be one — this could be a real production incident). You are asked to
report what is observably wrong, using the evidence channels above.
